## Redis 链路排障约定（含 BYPASS_REDIS 修复 2026-07-07）
- 登录"获取验证码"500 → 99% Upstash 链路问题（非阿里云短信）；阿里云失败是 503，节流命中 429，只有 Redis 异常是 500
- `BYPASS_REDIS=false` 时 sendOtp 必须连 Upstash（节流 set NX + 存码 setex，无 try/catch）；`BYPASS_REDIS=true` 兜底固定码 8888
- **BYPASS_REDIS=true 隐蔽陷阱**：原 `createRedisClient` 用 `maxRetriesPerRequest:null`（BullMQ 兼容），Upstash 不可达时 hget 等命令**永久 hang**（命令进重试队列永不 reject）→ getMe 挂死 → 前端"分身正在苏醒"永不消失
- **进程崩溃陷阱**：所有 ioredis 实例必须 `.on('error')`，否则 error 事件让 Node 崩溃；`duplicate()` 不继承监听器需单独加（live-redis.subscriber.ts 的 sub）；`subscribe()` 返回的 promise reject 也要 `.catch()`（Node 22 默认 unhandled-rejections=throw）
- **不能**用 `retryStrategy:()=>null` 修复 hang：连接进入 closed 状态 → `close` 事件 `flushQueue` 同步抛异常 → 进程崩
- **正确修复**（已落地）：BYPASS 模式 `maxRetriesPerRequest:1`（命令 ~60ms reject，不 hang）+ `lazyConnect:true` + `connectTimeout:1` + 保持默认 retryStrategy（连接留 reconnecting 不 close，不崩）；getMe 的 hget 额外加 2s `Promise.race` 超时兜底
- 改动文件：`redis/create-redis.ts` / `redis/redis.service.ts`(+.on error) / `live/live-redis.subscriber.ts`(+.on error +subscribe.catch) / `clones/clones.service.ts`(getMe 超时)
- QueueService 用独立 bullConnection（不受 BYPASS 影响），RedisService.client 仅业务用，可安全改 maxRetries
- 实锤：看 API 日志 ioredis 错误码（MAXREQUESTS/ECONNREFUSED/ETIMEDOUT）；Upstash 额度耗尽报 `max requests limit exceeded`
