## OTP 发送 500 排障诊断（仅诊断，未改代码）

用户报：登录页点"获取验证码"返回 Internal server error（500）。

### 根因
- `services/api/.env`：`BYPASS_REDIS=false` + `REDIS_URL=rediss://...upstash.io:6379`（Upstash 远程 Redis，TLS）
- `auth.service.ts` sendOtp 在非旁路模式下：①60s 节流 `redis.client.set(key,'1','EX',60,'NX')` ②存验证码 `redis.client.setex(...)` ——两步都打 Upstash
- 节流那步**无 try/catch 兜底**，Upstash 一旦连接/额度出问题 → reject 冒泡 → NestJS 转成 500
- 其他异常都不是 500：阿里云短信失败走 `ServiceUnavailableException`（503，sms.service 内部已 try/catch）；节流命中是 429 ——故 500 可锁定为 Redis 调用异常
- `.env` 注释明确写"Upstash 额度耗尽时临时启用 BYPASS_REDIS"，高度怀疑 Upstash 免费额度耗尽

### 旁路模式（BYPASS_REDIS=true）行为
- sendOtp 固定码 8888，不碰 Redis / 短信网关，直接返回
- loginWithOtp 也只校验 `code === '8888'`
- refresh/logout 跳过 JWT 黑名单检查
- 用途：Upstash 额度耗尽 / 远程 Redis 不可达时的临时兜底

### 验证方法（未执行）
看 API 进程 stdout/stderr 里 ioredis 抛的具体错误码：额度耗尽 → `MAXREQUESTS`/`command rate limit exceeded`；连接失败 → `ECONNREFUSED`/`ETIMEDOUT`/TLS 握手错误；认证错 → `NOAUTH`/`WRONGPASS`。

## 分身页"正在苏醒"加载不出修复（Redis hang + 进程崩溃）

### 现象
登录后点分身页，"分身正在苏醒…"两三分钟不消失。

### 根因（双重）
1. **API 进程崩了**（4000 端口无监听）：Upstash Redis 额度耗尽（500000/500002），ioredis error 事件无监听器 → Node 崩溃退出
2. **即使 API 在跑，getMe 也会 hang**：`createRedisClient` 的 `maxRetriesPerRequest:null`（BullMQ 兼容）+ Upstash 不可达 → `hget` 命令进重试队列**永久 pending** → getMe 挂死 → 前端"苏醒"永不消失

### 排错弯路
- 第一次试 `retryStrategy:()=>null`：命令 8ms reject（不 hang ✓），但连接进 closed 状态 → `close` 事件 `flushQueue` 同步抛异常 → 进程崩 ✗
- 发现 `live-redis.subscriber.ts` 的 `duplicate()` 实例没 `.on('error')`（duplicate 不继承监听器）→ 加上后 error 不崩，但 flushQueue 仍崩
- 最终方案：`maxRetriesPerRequest:1`（命令 60ms reject，不 hang）+ 保持默认 retryStrategy（连接留 reconnecting 不 close，不崩）

### 修复（4 文件）
- `redis/create-redis.ts`：BYPASS 模式 `maxRetriesPerRequest:1` + `lazyConnect:true` + `connectTimeout:1`（去掉 retryStrategy:null）
- `redis/redis.service.ts`：加 `.on('error')` Logger.warn 吞错误
- `live/live-redis.subscriber.ts`：duplicate 实例加 `.on('error')` + `subscribe().catch()`（Node 22 unhandled-rejections=throw）
- `clones/clones.service.ts`：getMe 的 hget 加 2s `Promise.race` 超时兜底（防御非 BYPASS 模式 Upstash 抖动）

### 验证
- 单元：hget 60ms reject（不 hang），连接 reconnecting（不 close 不崩）
- API 启动：编译 0 errors，"Nest application successfully started"，无 Redis 崩溃
- 端到端：带真实 token 调 /clones/me → 200，978ms（含 Redis hget，未 hang）；/clones/me/posts → 200，229ms，3 条帖子 isAutoGenerated 正常

### 教训
- `maxRetriesPerRequest:null` 是 BullMQ 专用，业务 client 不能用（命令会永久 hang）
- ioredis 所有实例（含 duplicate）必须有 error 监听器
- `retryStrategy:()=>null` 会触发 close→flushQueue 崩溃，不能用来"快速失败"
- Node 22 默认 unhandled-rejections=throw，subscribe 等 fire-and-forget 的 promise 也要 .catch()
