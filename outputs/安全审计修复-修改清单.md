# Echo 安全审计修复 — 修改清单

> 范围：拉黑 / 忽略 / 举报 三个用户控制权在 worker 与 API 读取路径中的全链路修复
> 验收：`cd services/api && npx tsc --noEmit` ✅ ｜ `cd ../worker && npx tsc --noEmit` ✅（均 exit 0）
> 约束遵守：未修改 Prisma schema、未修改客户端、BlockFilterService 可复用、每处修改均含中文注释

## 一、缺陷修复明细

### 缺陷1 — match-daily 不排除 block
**文件**：`services/worker/src/clone-runtime/match-bridge.ts`（`queryVectorCandidates`）

**问题**：pgvector 候选查询 WHERE 仅 `user_id != $2`，未排除被拉黑用户，导致 A 拉黑 B 后仍可能把 B 作为匹配候选推给 A（反之亦然）。

**修复**：在候选 SQL 中追加两条 NOT IN 子查询，复用 `$2`（seeker userId）做双向排除：
- `pe.user_id NOT IN (SELECT blocked_user_id FROM blocks WHERE blocker_user_id = $2)` —— 排除「seeker 拉黑的人」
- `pe.user_id NOT IN (SELECT blocker_user_id FROM blocks WHERE blocked_user_id = $2)` —— 排除「拉黑 seeker 的人」

不消耗新的参数位，原 `paramIdx` 逻辑不受影响。

---

### 缺陷2 — bridge 不过滤 dismissed/blocked
**文件**：`services/worker/src/clone-runtime/match-bridge.ts`（`bridgeMatchPushes`）

**问题**：`findMany({take:50, orderBy:{pushedAt:'desc'}})` 无状态过滤，`dismissed` 的 MatchPush 也被 bridge 成 session；且未过滤拉黑关系。

**修复**：
1. import 增加 `MatchPushStatus`；两条 findMany 路径的 where 均加 `status: MatchPushStatus.pending`，只 bridge 未处理的。
2. 拉取 pushes 后，收集涉及的全部 userId，一次性查询 `blocks` 表（`blockerUserId` 或 `blockedUserId` 命中任一），用排序后的无向对键 `${a}|${b}`（A|B 与 B|A 同 key）构建 blockedPairs 集合，内存过滤掉处于拉黑关系中的 push，再进入 session 创建循环。

---

### 缺陷3 — report-triage 不重审
**文件**：`services/worker/src/main.ts`（`report-triage` processor）

**问题**：举报帖子后仅把 `approved` 改回 `pending`，未重新入队 moderation，帖子实际不会被再次审核。

**修复**：在改回 pending 并写 `report.post_flagged` 审计后，按 worker 现有入队模式（与 post-draft worker 同款）使用原生 BullMQ Queue：
```ts
const modQ = new Queue('moderation', { connection });
await modQ.add('moderate', { postId: targetId });
await modQ.close();
```
并追加 `report.remoderate` 审计事件（"举报触发重审，已重新入队审核"）+ `logger.info`。

---

### 缺陷4 — API 读取路径未过滤 block
**新增基础设施**（可复用）：
- `services/api/src/common/block-filter.service.ts` — `BlockFilterService`
  - `getBlockedUserIds(userId)`：查 `blocks` 表 `OR(blockerUserId, blockedUserId)`，返回双向对端 userId 列表
  - `applyBlockFilter(prismaWhere, userId, field)`：合并双向 `notIn` 进任意 Prisma where（安全合并既有字段条件，不覆盖）
- `services/api/src/common/common.module.ts` — `@Global()` 模块，导出 `BlockFilterService`
- `services/api/src/app.module.ts` — 注册 `CommonModule`

**应用点**：
| 文件 | 方法 | 修复内容 |
|---|---|---|
| `feed/feed.controller.ts` | `list` / `one` | 把 `_userId` 改为 `userId` 并传入 service |
| `feed/feed.service.ts` | `list(userId,…)` / `getOne(userId,id)` | 注入 `BlockFilterService`；list 用 `clone.user.id notIn blockedIds` 过滤帖子；getOne 过滤帖子作者 + 评论作者（comments where），作者被拉黑则 404 |
| `sessions/sessions.service.ts` | `listForUser` | 注入 `BlockFilterService`；建 `cloneId→userId` 映射，过滤掉对端 userId 处于拉黑关系的 session（双向） |
| `matches/matches.service.ts` | `list` | 注入 `BlockFilterService`；用 `getBlockedUserIds` 替换原先仅排除「我拉黑的人」的单向过滤，改为双向；session 查询注释说明 candidate 已在上游双向过滤 |

> 注：原 matches 仅单向（`blockerUserId=userId`）排除，现在改为双向，符合"A 拉 B 后双向都不再匹配"的语义。

### 缺陷5 — 举报目标无白名单
**文件**：
- `services/api/src/reports/reports.dto.ts`：`targetType` 加 `@IsIn(['post','session','clone','user'])`，导出 `REPORT_TARGET_TYPES` 常量与 `ReportTargetType` 类型。
- `services/api/src/reports/reports.service.ts`：新增 `assertTargetExists(targetType, targetId)`，按类型查 `post`/`agentSession`/`digitalClone`/`user` 表，不存在抛 `NotFoundException`；`create` 开头调用该校验。

---

### 缺陷6 — 审核拒绝不通知
**文件**：
- `services/worker/src/clone-runtime/push.ts`：`WorkerPushType` 增加 `'moderation_rejected'`，`TITLE_MAP`/`BODY_MAP` 增加对应文案（标题"⚠️ 内容未通过审核"，正文"你的分身发布的内容未通过审核"）。
- `services/worker/src/main.ts`（`moderation` processor，`verdict==='unsafe'` 分支）：在审计后追加：
  - `sendPush(prisma, post.clone.userId, 'moderation_rejected', { postId })`（best-effort，`.catch(()=>{})`）
  - `publishLiveEvent(connection, { type:'feed', userId, payload:{ postId, moderationStatus:'rejected' } })` 通知前端刷新

## 二、附：阻塞验收的既有问题（最小修复）

**文件**：`services/worker/src/main.ts`（agent-turn worker，`history.map`）

**预存问题**：原代码 `role: (m.speakerCloneId === speakerId ? 'assistant' : 'user') as const` 对三元表达式使用 `as const`，触发 `TS1355`，导致 worker `tsc --noEmit` 长期失败（非本次缺陷引入，仅因缺陷6在其上方插入代码而行号下移）。

**最小修复**：改为 `as 'assistant' | 'user'` 字面量联合断言，语义等价、通过门禁。已加注释说明。
