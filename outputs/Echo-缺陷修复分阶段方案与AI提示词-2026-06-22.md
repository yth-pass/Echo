```
你是一个 NestJS 后端安全工程师。项目 Echo 的后端在 services/api（NestJS 11 + Prisma 6 + BullMQ + ioredis），入口模块是 src/app.module.ts。鉴权用 @nestjs/jwt，守卫在 src/common/jwt-auth.guard.ts，写 req.userId。审计服务在 src/audit/audit.service.ts，方法签名 audit.log(userId, eventType, summaryZh, referenceId?)。

修复：messages() 中取出 session 后，校验 myClone（通过 userId 查 clone）的 id 等于 session.cloneAId 或 session.cloneBId，否则抛 NotFoundException。

【缺陷4 - JWT_SECRET 弱默认值】app.module.ts 第 27 行 `secret: process.env.JWT_SECRET ?? 'dev-secret-change-me'`。
修复：启动时校验 process.env.JWT_SECRET 存在且长度 >= 32，否则抛错阻止启动。移除硬编码 fallback。

【缺陷5 - Feed/Posts 无鉴权】feed/feed.controller.ts 整个 controller 无 @UseGuards(JwtAuthGuard)。
修复：FeedController 加 @UseGuards(JwtAuthGuard)，从 req.userId 获取当前用户。

【缺陷6 - LLM proxy 无鉴权】llm/llm-proxy.controller.ts 仅 LlmProxyGuard（IP 限流），无 JwtAuthGuard。且 llm-proxy.guard.ts 读 req.user.id 但 JwtAuthGuard 写的是 req.userId。
修复：LlmProxyController 加 @UseGuards(JwtAuthGuard, LlmProxyGuard)；guard 中 req.user.id 改为 req.userId。

【缺陷7 - JWT 无失效机制】无 logout 接口，token 泄露后无法吊销。
修复：
- 新增 POST /auth/logout 接口：将当前 access token 的 jti（需在签发时注入）加入 Redis 黑名单，TTL = token 剩余有效期
- JwtAuthGuard 校验时先查 Redis 黑名单，命中则拒绝
- 签发 token 时 payload 加 jti: randomUUID()

【缺陷9 - 普遍无限流】posts、reports、onboarding 控制器无限流。
修复：这些 controller 加 @Throttle({ default: { limit: 20, ttl: 60000 } })（每分钟 20 次）。
```

**前置依赖**：阶段 1（鉴权守卫就位，req.userId 可用）

### AI 提示词

```
你是 Echo 项目的后端工程师。Echo 是 AI 分身社交产品，后端 services/api（NestJS+Prisma），异步 worker 在 services/worker（BullMQ）。Prisma schema 在 services/api/prisma/schema.prisma，其中 Block 模型有联合主键 (blockerUserId, blockedUserId)，MatchPush 模型有 status 字段（类型 MatchPushStatus，值 pending/dismissed/bridged）。worker 的 QueueService 在 services/api/src/queue/queue.service.ts，方法 enqueueModeration({postId})。

【缺陷1 - match-daily 不排除 block】services/worker/src/clone-runtime/match-bridge.ts 的 queryVectorCandidates 函数，WHERE 条件仅 `user_id != $2`，未排除被拉黑用户。
修复：在 pgvector 候选查询的 SQL 中加 `AND candidate_user_id NOT IN (SELECT blocked_user_id FROM blocks WHERE blocker_user_id = $1) AND candidate_user_id NOT IN (SELECT blocker_user_id FROM blocks WHERE blocked_user_id = $1)`（双向排除，因为 block 是单向的，A 拉 B 后 A 不应再收到 B，B 也不应被动收到 A）。

- services/api/src/feed/feed.service.ts 的 list() 和 getOne()：返回的帖子和评论需排除作者被当前用户拉黑、或拉黑了当前用户的

- services/api/src/matches/matches.service.ts 内查 session 的地方（约 46-67 行）：排除 block
修复：在这些查询中 JOIN Block 表过滤。用 Prisma 的 where 子句加 NOT EXISTS 或先查 block 列表再过滤。推荐先查当前用户的 block 列表（blockerUserId 或 blockedUserId 等于 userId 的所有对端 userId），然后用 `NOT in` 过滤。封装一个 BlockFilterService（services/api/src/common/block-filter.service.ts）提供 getBlockedUserIds(userId) 和 applyBlockFilter(prismaWhere, userId, field) 两个方法，避免重复代码。

【缺陷5 - 举报目标无白名单】services/api/src/reports/reports.dto.ts 的 targetType 是自由字符串。
修复：targetType 加 @IsIn(['post','session','clone','user']) 白名单校验；reports.service.ts 的 create 方法校验 targetId 在对应表中存在，不存在则抛 NotFoundException。
```

## 阶段 3：队列幂等与分身运行时健壮性

**目标**：修复 worker 侧的幂等性缺失和分身运行时的多个健壮性问题——agent-turn 重复消费、post-draft 绕过安全层、persona 注入、LLM 失败写空消息、pause 后任务仍执行。

**覆盖缺陷**：P0-8、P1（post-draft 绕过 composer、personaText 注入、LLM 失败写空消息、pause 不检查 status）、P2（forbiddenWords 无生成后校验）

**涉及文件**（services/worker/src/）：
- `main.ts`（agent-turn processor、moderation processor）
- `clone-runtime/scheduler.ts`
- `clone-runtime/llm.ts`
- `clone-runtime/boundaries.ts`
- `agent-platform/composer/prompt-composer.ts`
- `services/api/src/queue/queue.service.ts`（jobId 设置）

**前置依赖**：阶段 1（审计服务就位）

**验收标准**：
- [x] `enqueueAgentTurn` 设 jobId=`agent-turn:${sessionId}:${turnIndex}`
- [x] agent-turn processor 用 DB 唯一约束防重（turnId 字段）
- [x] `generatePostContent` 走 `composeSystemPrompt()`
- [x] persona 文本置于 boundaries 之后，且对 persona 做基础注入防护
- [x] `chat()` 返回 null 表示失败，agent-turn 不写空消息，标记 session 错误态
- [x] agent-turn processor 入口检查 clone.status，pause 则跳过
- [x] post-draft 生成后扫描 forbiddenWords，命中则不发帖并审计

### AI 提示词

```
你是 Echo 项目的 worker 工程师。worker 在 services/worker（TypeScript + BullMQ + Prisma），主入口 src/main.ts。分身运行时在 src/clone-runtime/，agent 平台（composer、affection）在 src/agent-platform/。LLM 调用封装在 src/clone-runtime/llm.ts 的 chat() 函数。prompt 组装在 src/agent-platform/composer/prompt-composer.ts 的 composeSystemPrompt()。队列服务在 services/api/src/queue/queue.service.ts。Prisma schema 中 AgentMessage 有 turnId 字段（@unique，可空），AgentSession 有 status 字段（active/completed/error），DigitalClone 有 status 字段（active/paused）。

【背景】审计发现 worker 存在幂等性缺失和运行时健壮性问题，可能导致消息重复、会话卡死、安全约束被绕过。

【缺陷1 - agent-turn 非幂等】services/api/src/queue/queue.service.ts 的 enqueueAgentTurn 调用 `.add('turn', payload)` 未设 jobId。worker main.ts 的 agent-turn processor 无幂等检查。重复消费导致消息重复、affinity 失真。
修复：
- queue.service.ts: enqueueAgentTurn 需接收 turnIndex 参数，设 jobId = `agent-turn:${sessionId}:${turnIndex}`，BullMQ 同 jobId 自动去重
- main.ts agent-turn processor 开头：用 upsert 写 AgentMessage（turnId = `${sessionId}-${turnIndex}`），若唯一约束冲突说明已处理，直接 return
- 入队前需先查当前 turnIndex（session 已有消息数），传给 enqueueAgentTurn

【缺陷2 - post-draft 绕过 composer】services/worker/src/clone-runtime/scheduler.ts 的 generatePostContent 手写 prompt 拼接 persona + boundary，未走 composeSystemPrompt()，未注入 safety/skill 层。而 agent-turn 正确走了 composer。
修复：generatePostContent 改用 composeSystemPrompt() 组装 system prompt（与 agent-turn 一致），user prompt 部分保留发帖场景的指令。boundary 信息通过 composer 的 L1 安全层注入，不再手动拼。

【缺陷3 - persona 注入】scheduler.ts 和 prompt-composer.ts 中用户编辑的 personaText 原样拼入 prompt，可被注入"忽略以上约束"。且 boundaries 在 persona 之后，可被覆盖。
修复：
```

```
修复：res 为 null 时不加 fallback 消息、不 +1，显示错误提示让用户重试。只有 res 成功才加消息和 +1。

【客户端缺陷3 - resetDialogueSession 失败掩盖】Onboarding.tsx 约 170-174,297-298 行，/dialogue/start 失败时 setDialogueReady(true) + return true。
修复：失败时 setDialogueReady(false)、setDialogueError(错误信息)、return false。不掩盖错误。

【客户端缺陷4 - 全量 refetch 竞态】Echo/src/App.tsx 约 162-171 行，handleLive 每条事件都 refreshFeed()，并发覆盖。
修复：
- 防抖：handleLive 用 200ms debounce 合并多个事件后一次性 refreshFeed
- in-flight 标记：loadFeed 加 ref 标记是否正在请求，请求中再触发则等当前完成后再请求一次（确保拿到最新）
- 或用 AbortController：新请求取消旧的

【客户端缺陷5 - 多 Tab WS】Echo/src/api/ws.ts + App.tsx，每 Tab 独立建连。
修复：用 BroadcastChannel 协调：
- 第一个 Tab 建 WS，后续 Tab 检测到已有 Tab 则不建连，通过 BroadcastChannel 接收消息
- WS Tab 关闭时，其他 Tab 选举一个接管
- 简化实现：BroadcastChannel 广播 live 事件，所有 Tab 都监听；WS 连接用一个 Tab 维护（用 localStorage lock 选举）

【客户端缺陷6 - 无 Router】Echo/src/App.tsx 全文，无路由。

【客户端缺陷7 - WS 无心跳】Echo/src/api/ws.ts 约 55-95 行。

【后端缺陷8 - finalize 跳步】services/api/src/onboarding/onboarding.service.ts 约 191-318 行，finalize 不校验 DIALOGUE_MIN_TURNS，可跳过对话；可重复调用。
```
